# contract-guard · SDD 契约前置守卫

## 这个 skill 是什么

你（主 Agent）以前直接被用户喊 `/speckit.specify`、`/speckit.plan`、`/speckit.tasks` 来做规格驱动开发（SDD）。
现在用户改喊 **contract-guard**。本 skill 的唯一职责是：

> **在你调用 spec-kit 之前，先从契约模板库 `lib/` 中匹配出与当前需求相关的「可验证契约模板」，把它们注入到需求描述里，再用这份「增强需求」去驱动 spec-kit。**

这样 spec-kit 生成的 spec/plan/tasks 从源头就带着可验证的下限，而不是事后补救。

> ⚠️ 注入的是**可验证的契约模板**（带前置条件 / 后置条件 / 不变式 / 验收标准），
> 不是"要小心 XX"这类模糊提醒。模糊提醒提不了下限，可验证契约才能。

---

## 核心原则（必须遵守，不要自作主张省略步骤）

1. **你不"代替"spec-kit 生成 spec**。你只负责注入契约 + 编排调用 spec-kit。生成 spec 永远由 `/speckit.specify` 完成。
2. **契约来自 `lib/`，不要凭空编造契约**。`lib/` 是经过治理审核的契约模板库，是可信源。匹配不到时如实说明，不要捏造一条契约塞进去。
3. **运行时只读本地 `lib/`，不联网**。版本升级由 `scripts/update.sh` 离线完成，与运行解耦。
4. **每一步都要回显进度**，让用户能看到"现在在做哪一步"，这是压低编排方差的关键。

---

## 执行流程（严格按顺序执行，每步做完回显一句状态）

### 第 0 步 · 加载契约模板库索引
读取 `lib/_index.json`。它定义了「契约分类」与「需求关键词 → 命中分类」的映射。
回显：`[contract-guard] 已加载契约模板库，共 N 个分类。`

### 第 1 步 · 需求分类
分析用户给出的需求描述，判断它涉及 `_index.json` 中的哪些分类。
- 同时用两种信号判断：① 关键词命中（来自 `_index.json` 的 `keywords`）；② 语义判断（即使没有字面关键词，只要语义涉及该分类也要命中）。
- 一个需求可命中多个分类。
回显：`[contract-guard] 需求分类完成，命中：幂等、并发、数据一致性。`

### 第 2 步 · 匹配契约模板
对每个命中的分类，`ref` 读取 `lib/` 下对应的 `.md` 文件，取出其中的契约模板条目。
- 一个分类文件里可能有多条契约，按"与当前需求的相关度"挑选适用的条目，不要全量塞入。
- 若所有分类都没命中：明确告诉用户"未匹配到契约模板"，并询问是否仍要继续走 spec-kit（此时退化为普通 SDD）。
回显：`[contract-guard] 已匹配 K 条契约模板。`

### 第 3 步 · 生成「增强需求」
把原始需求 + 匹配到的契约模板，拼成一份结构化的「增强需求包」，格式如下：

```
## 原始需求
<用户的原始需求原文>

## 必须满足的可验证契约（来自 contract-guard 契约模板库）
### 契约 1：<契约名>
- 前置条件：...
- 后置条件：...
- 不变式：...
- 验收标准：...
### 契约 2：...
```

回显：`[contract-guard] 已生成增强需求包，准备进入 spec-kit。`

### 第 4 步 · 编排调用 spec-kit（依次执行）
用上面的「增强需求包」作为输入，**依次**触发 spec-kit 的命令。每调用一个，必须等它产出后再进入下一个，并回显状态：

1. 调用 `/speckit.specify`，输入 = 增强需求包。
   - 完成后回显：`[contract-guard] specify 完成，进入 plan。`
2. 调用 `/speckit.plan`。
   - 完成后回显：`[contract-guard] plan 完成，进入 tasks。`
3. 调用 `/speckit.tasks`。
   - 完成后回显：`[contract-guard] tasks 完成，进入契约回查。`

> 注意：你是 spec-kit 命令的**调用方/编排方**，不是它的实现方。这些命令本身已由 spec-kit 提供。
> 若当前环境检测不到 spec-kit 命令，停止并提示用户："未检测到 spec-kit，请先安装：`uvx --from git+https://github.com/github/spec-kit.git specify init <project>`"。

### 第 5 步 · 契约回查（闭环验收）
spec-kit 产出 spec / plan / tasks 后，逐条核对**第 2 步注入的每一条契约**，是否在产出物中都有对应落点：
- 该契约的「验收标准」是否出现在 spec 的验收项 / tasks 的测试任务中？
- 若某条契约**未被覆盖**：明确列出"契约 X 的验收标准未在产出中体现"，并建议补充对应的 spec 验收项 / 测试任务。

输出一张回查表：

```
| 契约 | 是否覆盖 | 落点 / 缺口 |
|------|---------|------------|
| 幂等-支付提交 | ✅ | tasks#5 并发重放测试 |
| 并发-库存扣减 | ⚠️ 部分 | spec 提到，缺测试任务 |
```

回显：`[contract-guard] 契约回查完成，X 条已覆盖，Y 条存在缺口。`

---

## 已知局限（诚实告知用户）

- 第 4 步的编排是靠指令驱动主 Agent 完成的，属于软约束，存在漏调 / 乱序的可能。一旦发现 spec-kit 步骤被跳过，请重新喊 contract-guard。
- 契约库的完备性取决于 `lib/` 的沉淀程度，匹配不到不代表没有风险，只代表库里还没有这条契约。
- 契约库的更新由独立的治理项目（人工 review + LLM judge + 规则引擎）产出后 commit，本 skill 只消费、不生产契约。
