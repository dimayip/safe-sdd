# 时间与时区正确性契约（time-correctness）

> 领域：任何涉及时间戳、时区、定时调度、有效期/过期、时间差计算的系统（横切，不挑领域）。
> 原则：只否决"时间语义不明致跨时区/夏令时算错 / 用墙上时钟测时长被回拨毁掉 / 边界时刻判定错 / 定时任务漂移或重复触发"，不绑定具体时区库或时钟源。

## 契约 TIME-1：时间存储与传输语义明确，跨时区不产生歧义
- 前置条件：系统存储、传输或对外暴露时间值（落库、API 字段、日志、消息）。
- 后置条件：每个时间值都有明确且一致的语义——要么是带时区的绝对时刻（UTC/带 offset），要么显式声明为"本地墙上时间 + 所属时区"；不存在裸的、无时区信息且语义靠猜的时间字符串跨边界传递。
- 不变式：同一时刻在生产端写入与消费端读出，换算回绝对时刻后相等；不因服务器时区、客户端时区或序列化默认时区不同而产生偏移。
- 验收标准：
  - 用例：东八区写入、UTC 服务器读取同一条记录 → 还原出的绝对时刻一致，不差 8 小时。
  - 用例：API 返回时间字段 → 带时区信息（如 ISO-8601 带 Z 或 offset），消费方无需猜测即可正确解析。

## 契约 TIME-2：时长/超时测量用单调时钟，不受墙上时钟跳变影响
- 前置条件：代码测量"经过了多久"——超时判定、限流窗口、重试退避、性能耗时、缓存 TTL 计时。
- 后置条件：时长测量基于单调时钟（monotonic），不直接相减两个墙上时钟（wall-clock）值；NTP 校时、夏令时切换、人工改系统时间导致的时钟回拨/跳变，不会让时长变负、超时提前触发或永不超时。
- 不变式：对同一段真实流逝的物理时间，测得的时长恒为非负且单调，不随系统时钟被外部调整而突变。
- 验收标准：
  - 用例：计时过程中将系统时钟向后回拨 → 测得时长不为负、不立即判超时。
  - 用例：跨夏令时切换点测一段 1 小时的真实耗时 → 结果约为 1 小时，不出现 0 或 2 小时。

## 契约 TIME-3：时间边界与特殊时刻判定正确，无差一与歧义时刻漏洞
- 前置条件：存在基于时间的边界判定——有效期/过期、生效起止、按天/按月归集、定时窗口的开闭区间。
- 后置条件：边界开闭语义明确并实现一致（如"含起不含止"贯穿到底）；正确处理夏令时切换日（缺失/重复的本地时刻）、月末/闰年/闰日、跨午夜归属；不出现差一天、差一秒或边界时刻被双重/零次计入。
- 不变式：任一事件按时间边界归类时，恰好落入一个区间，既不重复计入也不遗漏；过期判定在边界时刻的行为与声明语义一致。
- 验收标准：
  - 用例：有效期到 23:59:59 的优惠 → 在该秒及之后的判定结果符合声明的开闭语义，不早失效也不晚失效一整天。
  - 用例：夏令时"回拨"导致本地 01:30 出现两次 → 落在该时刻的事件不被重复统计或错配日期。

## 契约 TIME-4：定时与调度准时且不漂移、不重复、不丢触发
- 前置条件：存在周期性或定点调度——cron、延时队列、定时刷新、心跳、续约。
- 后置条件：调度按声明的频率触发，不因单次执行超时而累积漂移或并发重入；时钟跳变、补偿追赶（catch-up）、多实例部署下不产生重复触发或整窗丢触发；关键定时任务具备错过补偿或幂等保护。
- 不变式：在声明周期内，每个调度窗口被触发的次数符合预期（恰好一次或幂等等价于一次），不因时钟调整或多副本而偏离。
- 验收标准：
  - 用例：单次任务执行耗时超过周期 → 不出现并发重入或后续触发时刻持续漂移。
  - 用例：多实例同时部署同一 cron → 通过分布式锁/选主保证每个窗口仅有效执行一次（依赖 IDEM-1 / 防重）。

---
> 关联契约：与 failure-retry（超时退避用单调时钟）、storage-durability（TTL 计时）、realtime-iot（时钟漂移/新鲜度）、observability（时间戳对齐）互补；本契约聚焦"时间语义与时钟正确性"这一横切维度。
