# 存储持久化与缓存一致性契约（storage-durability）

> 领域：数据库 / 缓存 / 对象存储 / 文件系统等持久化层。
> 原则：只否决"声明已持久化实则可丢 / 缓存与源长期不一致 / 写入非原子致半截数据 / 磁盘耗尽无保护"，不绑定 MySQL/Redis/S3 等具体产品。

## 契约 SD-1：持久化承诺与落盘语义一致
- 前置条件：写操作向调用方返回"成功/已提交"，调用方据此认为数据不会丢。
- 后置条件：返回成功前，数据已落到声明的持久化级别（如 WAL fsync / 多副本确认）；进程崩溃、断电后已确认的写不丢失。
- 不变式：已向调用方确认成功的写 ⇒ 重启后可读到（不弱于声明的持久化级别）。
- 验收标准：
  - 用例：写返回成功后立即 kill 进程并重启 → 数据仍可读出。
  - 用例：若采用异步刷盘（性能换持久性），必须在接口语义中显式声明可能丢失的窗口，不得谎称强持久。

## 契约 SD-2：缓存与数据源最终一致且无脏读永驻
- 前置条件：读路径前置缓存，写路径更新数据源（旁路缓存/读写穿透）。
- 后置条件：写后缓存按声明策略（失效/更新）处理；并发读写下不产生"旧值长期驻留缓存"；缓存未命中能回源且防击穿/雪崩。
- 不变式：任一缓存项与数据源的不一致窗口 ≤ 声明 TTL/失效时延；不存在源已改、缓存永不更新的项。
- 验收标准：
  - 用例：更新源后并发读 → 在声明时延内读到新值，不长期返回旧值。
  - 用例：热点 key 失效瞬间高并发 → 通过单飞/互斥回源，不击穿数据源。

## 契约 SD-3：写入原子性，不留半截数据
- 前置条件：一次逻辑写涉及多字段/多文件/多对象（如先写数据再写索引/元数据）。
- 后置条件：要么全部生效，要么全部不生效；中途失败不留下可被读到的半成品（部分写、孤儿文件、悬挂索引）。
- 不变式：任意时刻对外可见状态都是"完整逻辑写"的边界，不存在中间态被当作有效数据读出。
- 验收标准：
  - 用例：多步写在中间步骤注入失败 → 已写部分被回滚/不可见，无孤儿数据。
  - 用例：上传大对象中断 → 不暴露半截对象，重传幂等覆盖（依赖 IDEM-1）。

## 契约 SD-4：容量与配额有保护，耗尽不静默损坏
- 前置条件：存储容量、inode、连接数、配额有上限，可能被写满。
- 后置条件：接近上限时有告警/限流/拒写策略；写满时返回明确错误而非静默截断、覆盖或损坏既有数据。
- 不变式：容量耗尽只影响新写入（明确失败），不破坏已持久化的存量数据。
- 验收标准：
  - 用例：写到容量上限 → 新写返回明确的容量错误，存量数据完好可读。
  - 用例：达到配额阈值 → 触发告警/降级，不直接静默丢弃数据。
