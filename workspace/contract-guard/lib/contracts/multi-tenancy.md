# 多租户与资源隔离契约（multi-tenancy）

> 领域：SaaS / 云平台 / 共享基础设施（跨大型工程与分布式）。
> 原则：只否决"租户数据串台 / 配额失控 / 噪声邻居拖垮全局"，不绑定隔离实现（独库/共库分 schema/行级）。

## 契约 MT-1：租户数据严格隔离（不串台）
- 前置条件：多个租户共用同一套服务/存储，数据按 tenant_id 区分。
- 后置条件：任一查询/写入/缓存/导出都被强制限定在当前租户作用域内，A 租户无法读到或改到 B 租户数据（呼应 SEC-2 越权）。
- 不变式：每次数据访问的有效作用域 ⊆ 当前请求租户；不存在缺失 tenant 过滤的查询路径。
- 验收标准：
  - 用例：构造缺省 tenant 条件的查询 → 被框架强制注入租户过滤或拒绝，不返回跨租户数据。
  - 用例：缓存/导出/日志 → 不混入其他租户数据。

## 契约 MT-2：资源配额可限制（防失控）
- 前置条件：租户消耗共享资源（存储、QPS、计算、连接数）。
- 后置条件：每租户有配额上限，超额请求被限流/拒绝并明确告知，不允许单租户无限占用（呼应 SEC-3）。
- 不变式：单租户资源占用 ≤ 其配额；总占用 ≤ 系统容量。
- 验收标准：
  - 用例：单租户超额调用 → 被限流，返回配额超限错误。
  - 用例：所有租户配额之和的承诺与系统容量一致，不超售导致整体不可用。

## 契约 MT-3：噪声邻居隔离（故障/高负载不外溢）
- 前置条件：单租户的异常流量或慢请求可能影响同实例其他租户。
- 后置条件：通过资源隔离（线程池/连接池/优先级/限流分桶）使单租户的高负载或故障不击穿其他租户的可用性（呼应 PT-3 背压、FR-3 降级）。
- 不变式：单租户故障的影响半径被限制在该租户内。
- 验收标准：
  - 用例：单租户打满其配额并持续慢请求 → 其他租户 P99 不被显著拖累。
  - 用例：单租户触发异常 → 不导致共享实例整体崩溃。
