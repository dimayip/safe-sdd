# 音视频传输与实时通信契约（media-streaming）

> 领域：直播 / RTC / 点播 / 音视频会议。
> 原则：只否决"协商失败无降级 / 丢包致崩溃或卡死 / 音画失同步 / 码率不自适应致卡顿"，不绑定 WebRTC/RTMP/HLS 等具体协议。

## 契约 MS-1：编解码与能力协商有兜底
- 前置条件：收发两端的编解码器、分辨率、帧率支持范围可能不一致。
- 后置条件：通过协商取双方支持的交集；无可用交集时降级到基线能力（如 H.264 baseline）或明确失败提示，绝不在不兼容编码上强行传输致黑屏/花屏。
- 不变式：实际使用的编解码参数 ∈ 收发双方能力的交集。
- 验收标准：
  - 用例：一端不支持对端首选编码 → 协商落到双方都支持的编码，画面正常。
  - 用例：无任何交集 → 返回明确的不兼容错误，不进入花屏状态。

## 契约 MS-2：丢包与抖动下不崩溃且可恢复
- 前置条件：实时传输（UDP 类）存在丢包、乱序、抖动。
- 后置条件：通过抖动缓冲、NACK/FEC/重传在声明的丢包率上限内维持可用音视频；超限时优雅降质（降分辨率/丢帧到关键帧）而非崩溃或永久卡死。
- 不变式：在声明丢包率上限内播放不中断；解码器遇坏帧不崩溃，能在下个关键帧恢复。
- 验收标准：
  - 用例：注入 5% 丢包 → 画面有降质但持续播放，不卡死。
  - 用例：丢失非关键帧 → 在下一个 I 帧自动恢复，无永久花屏。

## 契约 MS-3：音画同步在容差内
- 前置条件：音频与视频分轨传输，各有独立时间戳与缓冲。
- 后置条件：基于统一时间基准（如 RTP 时间戳 / PTS）对齐音视频，唇音同步偏差控制在声明容差内（如 ±80ms）。
- 不变式：|音频呈现时间 − 视频呈现时间| ≤ 声明同步容差。
- 验收标准：
  - 用例：视频解码慢于音频 → 通过丢帧/等待将偏差拉回容差内。
  - 用例：长时间播放 → 音画不累积漂移超出容差。

## 契约 MS-4：码率自适应跟随网络
- 前置条件：网络带宽实时波动（弱网、拥塞）。
- 后置条件：带宽估计驱动自适应码率（ABR），带宽下降时主动降码率/降分辨率保流畅，回升时再提质；切换平滑不致频繁抖动（hysteresis）。
- 不变式：发送码率 ≤ 估计可用带宽 × 安全系数；不持续发送超过链路容量的数据致拥塞崩溃。
- 验收标准：
  - 用例：带宽骤降 → 在声明时间内降到匹配码率，卡顿率不超阈值。
  - 用例：带宽恢复 → 平滑提质，不在两档间反复横跳。
