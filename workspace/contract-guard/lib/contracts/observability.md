# 可观测性契约（observability）

> 领域：监控 / 告警 / 日志 / 链路追踪 / 指标。
> 原则：只否决"关键路径无埋点 / 告警漏报或风暴 / 日志泄敏或丢失关键上下文 / 监控自身拖垮业务"，不绑定 Prometheus/OpenTelemetry/ELK 等具体栈。

## 契约 OB-1：关键路径可观测且可定位
- 前置条件：系统存在影响 SLA 的关键链路（下单、支付、核心接口）。
- 后置条件：关键路径埋有指标（RED：速率/错误/时延）、结构化日志与可串联的 trace-id；出现故障时能据此定位到具体环节，而非只知"出错了"。
- 不变式：每个关键请求都带可贯穿上下游的唯一关联 ID；该 ID 在日志/指标/trace 中可对齐。
- 验收标准：
  - 用例：注入一次失败请求 → 能通过 trace-id 还原完整调用链与失败节点。
  - 用例：核心接口 → 有 P50/P95/P99 时延与错误率指标，且可按维度下钻。

## 契约 OB-2：告警有效，不漏报不风暴
- 前置条件：基于指标/日志配置告警规则，存在抖动、批量故障等噪声源。
- 后置条件：真实故障必告警（不漏报）；同源大量告警经聚合/抑制/去重收敛（不风暴）；告警带可操作信息（影响面、处置指引）。
- 不变式：单一根因产生的告警在抑制后条数有上界；SLA 突破必触发对应告警。
- 验收标准：
  - 用例：核心指标突破阈值 → 在声明时延内触发告警，不漏报。
  - 用例：一次依赖故障引发批量错误 → 告警被聚合为有限条，不刷屏淹没值班。

## 契约 OB-3：日志安全合规且保留关键上下文
- 前置条件：日志中可能混入敏感信息（密码、令牌、身份证、卡号）。
- 后置条件：敏感字段在落盘/输出前脱敏或屏蔽；同时保留排障必需的上下文（请求 ID、关键参数摘要），不因脱敏过度而丧失可定位性。
- 不变式：持久化日志中不出现明文敏感数据；关键链路上下文不被全量丢弃。
- 验收标准：
  - 用例：请求含密码/令牌 → 日志中对应字段被屏蔽，不可还原。
  - 用例：脱敏后 → 仍能通过保留的关联 ID 串起整条链路。

## 契约 OB-4：监控开销有界，不反噬业务
- 前置条件：埋点、采样、日志写入本身消耗 CPU/内存/IO 与下游存储。
- 后置条件：采集走异步/采样/限流，监控背压或后端不可用时降级（丢监控数据而非阻塞业务），监控开销控制在声明预算内。
- 不变式：监控子系统故障或拥塞 ⇒ 不阻塞、不拖垮主业务路径。
- 验收标准：
  - 用例：监控后端宕机 → 业务请求正常返回，监控数据按策略丢弃/缓冲，不阻塞。
  - 用例：高峰期 → 采样/限流生效，埋点开销不超声明预算。
