# 大数据批流处理契约（data-pipeline）

> 领域：大数据 / 流处理 / ETL / 数仓。
> 原则：只否决"重复计算 / 迟到数据被错算 / 流批口径不一致 / 状态丢失"，不绑定 Flink/Spark/Kafka 等引擎。

## 契约 DP-1：端到端处理语义明确（精确一次效果）
- 前置条件：数据从源经过处理写入汇（sink），中途可能因故障重启、重放。
- 后置条件：声明并保证端到端语义（at-least-once + 幂等汇 ⇒ exactly-once-effect）；重启/重放后汇中结果不重复、不丢失（依赖 IDEM-1、MSG-1）。
- 不变式：对同一输入数据，无论失败重试多少次，汇中对应输出至多体现一次。
- 验收标准：
  - 用例：处理中途 kill 并重启 → 汇中无重复记录、无丢失记录。
  - 用例：源端重放一段数据 → 聚合结果不被重复累加。

## 契约 DP-2：迟到与乱序数据有确定处理策略
- 前置条件：基于事件时间（event-time）做窗口聚合，数据可能乱序或迟到。
- 后置条件：定义水位线（watermark）与允许迟到上限；窗口内迟到数据按声明策略（更新结果/进侧输出/丢弃）处理，绝不静默错算。
- 不变式：超过允许迟到上限的数据被显式路由（侧输出/丢弃计数），不悄悄混入已关闭窗口。
- 验收标准：
  - 用例：注入迟到数据 → 在允许范围内更新对应窗口结果；超限则进侧输出且被计数。
  - 用例：乱序到达 → 按事件时间归入正确窗口，不按到达时间错分。

## 契约 DP-3：流批结果口径一致
- 前置条件：同一指标既有实时流计算，又有离线批回算（lambda/kappa 混合）。
- 后置条件：流与批对同一时间窗口、同一口径的计算结果在容差内一致；以批回算结果为准做校正时收敛。
- 不变式：流批同口径指标差异 ≤ 声明容差。
- 验收标准：
  - 用例：对同一天数据，流结果 vs 批回算结果 → 差异在容差内。

## 契约 DP-4：状态有界且可恢复
- 前置条件：有状态算子（聚合、join、去重）的状态随 key 基数增长。
- 后置条件：状态有 TTL 或清理策略防止无限膨胀；checkpoint/savepoint 可恢复，恢复后从一致位点续算。
- 不变式：状态规模有上界；恢复点与数据位点一致，不丢不重。
- 验收标准：
  - 用例：长时间运行 → 状态因 TTL 回收，不无限增长致 OOM。
  - 用例：从 checkpoint 恢复 → 续算结果与未中断一致。
