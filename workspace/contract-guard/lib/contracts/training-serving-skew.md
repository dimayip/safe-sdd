# 训练-服务一致性契约（training-serving-skew）

> 领域：AI 训推 / 机器学习系统。
> 原则：只否决"线上特征与训练特征不同源导致效果劣化 / 不可复现 / 标签穿越"，不绑定特征平台实现。

## 契约 TSS-1：训练与在线特征同源同逻辑
- 前置条件：模型训练用的特征与线上推理用的特征由各自管线产生。
- 后置条件：同一特征在训练侧与服务侧的计算逻辑、口径、单位一致；差异（skew）被持续监控且在阈值内。
- 不变式：对同一输入样本，训练特征值与线上特征值在容差内相等。
- 验收标准：
  - 用例：抽样比对训练/线上特征 → skew 比例低于声明阈值，超阈告警。
  - 用例：特征逻辑变更 → 训练与服务侧同步更新，不允许单侧改动上线。

## 契约 TSS-2：禁止标签/未来信息穿越
- 前置条件：训练样本的特征基于某时间点之前的数据构造。
- 后置条件：任一特征只使用样本事件时间点 T 之前可获得的信息，绝不使用 T 之后（含标签）的数据。
- 不变式：特征时间戳 < 标签/预测时间戳，无未来信息泄漏。
- 验收标准：
  - 用例：对特征做时间穿越检测 → 无任何特征引用 T 之后数据。
  - 用例：离线评估 AUC 与上线效果差距在合理范围（无"离线虚高"穿越信号）。

## 契约 TSS-3：模型与数据可复现
- 前置条件：模型由特定数据集 + 代码 + 随机种子 + 超参训练得到。
- 后置条件：给定相同数据快照、代码版本、种子与环境，可复现等价模型；模型产物与其训练输入（数据版本/代码版本）强绑定可追溯。
- 不变式：每个上线模型版本 → 唯一可追溯的训练输入指纹。
- 验收标准：
  - 用例：用记录的训练配置重训 → 指标在容差内一致。
  - 用例：任意线上模型 → 可定位其数据版本与代码 commit。
