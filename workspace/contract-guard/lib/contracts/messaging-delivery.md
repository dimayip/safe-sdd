# 消息投递语义契约（messaging-delivery）

> 领域：分布式计算 / 消息队列 / 事件驱动。
> 原则：只否决"消息丢失 / 重复处理产生重复副作用 / 顺序错乱导致状态错误"，不绑定 MQ 实现。

## 契约 MSG-1：投递语义明确且消费端幂等
- 前置条件：系统通过消息/事件传递状态变更，存在至少一次（at-least-once）投递可能。
- 后置条件：消费端对同一消息重复消费 N 次的最终效果等同于消费一次（消费幂等，依赖 IDEM-1）；声明的投递语义（at-least-once / exactly-once-effect）可被验证。
- 不变式：每条消息的业务副作用至多发生一次，即使被重复投递。
- 验收标准：
  - 用例：同一消息重投 100 次 → 业务副作用计数为 1。
  - 用例：消费成功但 ack 丢失导致重投 → 不产生第二次副作用。

## 契约 MSG-2：消息不丢失（可达性）
- 前置条件：消息从生产到消费跨网络与 broker。
- 后置条件：生产确认成功的消息最终一定被消费或进入死信可追溯，不存在"确认成功却凭空消失"。
- 不变式：生产确认 → 必然有消费或死信落点。
- 验收标准：
  - 用例：注入 broker 重启/消费者崩溃 → 已确认消息不丢失，恢复后被消费或进入 DLQ。

## 契约 MSG-3：需要顺序的场景顺序可保证
- 前置条件：存在对同一实体有先后依赖的事件（如"创建→更新→删除"）。
- 后置条件：同一分区键的事件按生产顺序被处理，乱序到达时能被检测并纠正（版本/序号）。
- 不变式：对同一实体的状态变更不会被旧事件错误覆盖。
- 验收标准：
  - 用例：乱序投递"更新(v2)、创建(v1)" → 最终状态为 v2，不被 v1 回退。
