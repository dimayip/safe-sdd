# 失败与重试契约（failure-retry）

> 适用：任何依赖外部 IO（网络/DB/第三方）的调用。
> 原则：只否决"失败被吞 / 重试放大故障 / 无降级"，不绑定重试库或熔断器实现。

## 契约 FR-1：失败必须可见且分类
- 前置条件：调用可能因超时、限流、依赖不可用等原因失败。
- 后置条件：失败不被静默吞掉；区分"可重试错误（超时/瞬时）"与"不可重试错误（参数非法/业务拒绝）"，并据此决定后续动作。
- 不变式：每个失败都有结构化记录（错误类型、是否可重试、上下文），可被监控捕获。
- 验收标准：
  - 用例：依赖超时 → 标记为可重试并按策略重试；参数非法 → 直接失败不重试。
  - 用例：任意失败路径 → 有日志/指标可观测，无"吞异常返回成功"。

## 契约 FR-2：重试安全且有上界，不放大故障
- 前置条件：对可重试错误执行重试。
- 后置条件：重试操作本身幂等（依赖 IDEM-1）；重试次数有上限；采用退避（backoff）避免雪崩；达到上限后进入降级或明确失败。
- 不变式：重试不会产生重复副作用；重试流量在退避下不会压垮已不稳定的依赖。
- 验收标准：
  - 用例：依赖抖动 → 重试在退避下收敛，副作用仍唯一。
  - 用例：依赖持续不可用 → N 次后停止重试并触发降级/熔断，不无限重试。

## 契约 FR-3：关键路径有降级兜底
- 前置条件：存在用户可感知的关键链路依赖非核心服务。
- 后置条件：非核心依赖故障时，主链路降级返回兜底结果而非整体不可用。
- 不变式：核心功能可用性不被非核心依赖的故障直接击穿。
- 验收标准：
  - 用例：推荐/营销等旁路依赖宕机 → 主流程返回兜底，仍可完成核心操作。
