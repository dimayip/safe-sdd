# 推荐系统服务质量契约（recommendation-serving）

> 领域：视频流 / 信息流推荐系统。
> 原则：只否决"无候选时白屏 / 重复曝光刷屏 / 新鲜度失效 / 召回链路故障无兜底"，不绑定召回排序实现。

## 契约 REC-1：始终有可返回结果（冷启动 + 兜底）
- 前置条件：个性化召回可能返回空（新用户、无画像、召回服务故障）。
- 后置条件：任何情况下推荐接口都返回非空且合规的结果——个性化为空时降级到热门/兜底池，绝不返回白屏。
- 不变式：返回列表长度 ≥ 声明的最小条数，或返回明确的兜底内容。
- 验收标准：
  - 用例：新用户无画像 → 返回冷启动/热门兜底，非空。
  - 用例：召回服务超时/宕机 → 降级兜底返回（依赖 FR-3）。

## 契约 REC-2：曝光去重与疲劳控制
- 前置条件：用户在短时间内多次刷新/翻页。
- 后置条件：同一用户在声明窗口内不重复曝光相同内容（除非显式置顶）；对已多次曝光未点击的内容做疲劳降权。
- 不变式：单用户单会话内同一 item 曝光次数 ≤ 声明上限。
- 验收标准：
  - 用例：连续翻页 N 次 → 无重复 item 刷屏，去重生效。
  - 用例：反复曝光未点击 item → 被疲劳降权，不持续霸屏。

## 契约 REC-3：新鲜度与实时性 SLA
- 前置条件：推荐效果依赖近实时的用户行为反馈与内容时效。
- 后置条件：实时特征/行为反馈在声明的时延上界内生效；过期内容（如已下架视频）不被推荐。
- 不变式：被推荐内容均处于"可播放/未下架"有效状态；实时反馈生效时延 ≤ SLA。
- 验收标准：
  - 用例：内容下架后 → 在声明时延内从推荐结果中消失。
  - 用例：用户刚产生负反馈 → 在 SLA 内不再推同类，不滞后过久。

## 契约 REC-4：多样性与意图泄漏防护
- 前置条件：纯按点击率排序易导致同质化堆叠与信息茧房。
- 后置条件：结果满足声明的多样性约束（类目/作者去集中）；排序特征不使用本次未发生的未来行为（防穿越，呼应 TSS-2）。
- 不变式：单屏结果的类目/作者集中度不超过声明阈值。
- 验收标准：
  - 用例：单屏结果 → 同一作者/类目占比不超过阈值。
