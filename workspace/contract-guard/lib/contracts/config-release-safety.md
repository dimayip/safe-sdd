# 配置变更与发布安全契约（config-release-safety）

> 领域：大型软件工程 / 持续交付 / 运维。
> 原则：只否决"配置/发布一改就全量炸 / 不可回滚 / 模块依赖成环"，不绑定 CI/CD 工具。

## 契约 CRS-1：变更可灰度、可回滚
- 前置条件：配置或代码变更将影响线上行为。
- 后置条件：变更支持灰度放量（小流量先行）并保留一键回滚到上一稳定版本；回滚后系统状态恢复到变更前等价状态。
- 不变式：任意已发布变更都有对应的、可执行的回滚路径。
- 验收标准：
  - 用例：灰度发现异常 → 在声明时间内回滚，影响面限于灰度流量。
  - 用例：配置错误推送 → 可快速回退，不需重新构建发布全量。

## 契约 CRS-2：配置变更前置校验
- 前置条件：配置由人工或系统下发，可能含非法值。
- 后置条件：配置在生效前经过 schema/取值域校验，非法配置被拒绝且不影响当前运行实例（依赖 BN-1）。
- 不变式：生效中的配置始终是已通过校验的合法配置。
- 验收标准：
  - 用例：下发越界/格式错误配置 → 被拒绝，线上沿用上一份合法配置。

## 契约 CRS-3：模块依赖方向无环
- 前置条件：系统由多模块/多服务组成，存在相互依赖。
- 后置条件：依赖关系为有向无环（DAG），高层依赖低层，不出现循环依赖导致的构建/启动死锁与变更扩散。
- 不变式：模块依赖图无环。
- 验收标准：
  - 用例：依赖分析 → 无循环依赖；新增依赖若成环则 CI 拦截。
