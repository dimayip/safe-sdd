# 资金与对账契约（financial-reconciliation）

> 领域：电商交易 / 支付 / 账务。
> 原则：只否决"资金凭空增减 / 账实不平 / 支付与订单状态错位"，不绑定支付通道与账务实现。

## 契约 FIN-1：资金守恒，账实可勾稽
- 前置条件：一笔交易涉及多方资金流转（买家、平台、商家、渠道）。
- 后置条件：任意时刻所有账户余额变动之和为零（除显式入金/出金）；业务订单金额与账务流水、与渠道对账单三方可勾稽平衡。
- 不变式：Σ各账户变动 = 0；订单金额 = Σ对应账务分录。
- 验收标准：
  - 用例：日终对账 → 业务库、账务库、渠道账单三方一致，差异为 0 或落入可追溯差异表。
  - 用例：注入任意中间失败 → 不出现资金凭空增加或消失。

## 契约 FIN-2：支付与订单状态严格对应（不漏单/不重复扣款）
- 前置条件：支付结果通过异步回调/主动查询获得，可能重复、乱序、丢失。
- 后置条件：支付成功 ⇔ 订单置为已支付，二者最终严格一致；回调重复不导致重复发货/重复入账（依赖 IDEM-1、MSG-1）；回调丢失由对账补偿兜底。
- 不变式：不存在"已扣款未建单"或"已发货未收款"的永久错位。
- 验收标准：
  - 用例：支付回调重复 100 次 → 订单只置已支付一次，账务只记一次。
  - 用例：回调丢失 → 主动查询/对账在声明窗口内补偿订单状态。

## 契约 FIN-3：价格与优惠计算确定且可重算
- 前置条件：最终应付 = 商品价 - 优惠 - 券 + 运费 + 税，多因子组合。
- 后置条件：相同下单上下文重复计算结果一致（确定性）；优惠不可叠加超过商品价导致负应付；计算过程可审计重放。
- 不变式：最终应付金额 ≥ 0；同一上下文计算结果唯一。
- 验收标准：
  - 用例：叠加多券致总优惠 > 原价 → 被规则拦截，应付不为负。
  - 用例：用下单快照重算金额 → 与下单时一致。
