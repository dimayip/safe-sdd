# 资源生命周期与泄漏契约（resource-lifecycle）

> 领域：任何持有有限资源的系统——连接/句柄/文件描述符/线程/协程/锁/内存/临时文件/订阅（横切，不挑领域）。
> 原则：只否决"资源只获取不释放致泄漏耗尽 / 异常路径漏释放 / 关闭后仍被使用 / 进程退出不优雅丢在途请求"，不绑定具体语言的资源管理机制。

## 契约 RES-1：资源获取与释放成对，异常路径也不泄漏
- 前置条件：代码获取了有限资源（连接、文件句柄、锁、协程/线程、内存缓冲、订阅/监听、临时文件）。
- 后置条件：无论正常返回、提前 return、还是抛异常/被取消，已获取的资源都在其生命周期结束时被确定性释放（defer/try-finally/RAII/with 等机制保证）；不存在"成功路径释放、异常路径漏释放"的分叉。
- 不变式：在任意时刻，已获取未释放的资源数量 = 当前确有正当用途的数量；长期运行下资源占用不单调增长。
- 验收标准：
  - 用例：在持有资源期间注入异常/取消 → 资源仍被释放，计数回落，不泄漏。
  - 用例：高频获取-释放循环长时间运行 → 句柄数/连接数/内存稳定在上限内，不持续爬升直至耗尽。

## 契约 RES-2：池化资源借还守恒，借不到时有界等待而非无限堆积
- 前置条件：使用资源池（连接池、线程池、对象池），容量有上限。
- 后置条件：借出的资源最终归还（含异常路径）；池满时调用方按声明策略处理（有界排队 + 超时 / 拒绝 / 降级），不无限阻塞也不无限扩容；归还的资源状态被重置，不携带上次使用的脏状态。
- 不变式：池中"空闲 + 借出 = 总容量"恒成立；不存在借出后永不归还的泄漏项导致池被逐渐耗干。
- 验收标准：
  - 用例：借用方在使用中崩溃/超时 → 资源被回收或超时强制归还，池不被逐渐抽干。
  - 用例：并发请求数超过池容量 → 多余请求有界等待并在超时后明确失败，不无限堆积导致雪崩。

## 契约 RES-3：资源关闭后不再被使用，无 use-after-close / double-free
- 前置条件：资源可被关闭/释放，且可能被多处引用或并发访问。
- 后置条件：资源关闭后再对其读写/操作会被安全拒绝（明确错误），不读到脏数据或触发崩溃；同一资源不被释放两次；关闭与使用之间的并发竞态被正确同步。
- 不变式：任一资源句柄在其生命周期内最多被释放一次；关闭状态对所有引用者可见，关闭后无有效操作发生。
- 验收标准：
  - 用例：一处关闭连接后另一处继续使用 → 返回明确的"已关闭"错误，不发送到已回收的连接上。
  - 用例：并发触发对同一资源的两次关闭 → 仅释放一次，无 double-free / panic。

## 契约 RES-4：进程/服务优雅停机，在途请求与资源被妥善收尾
- 前置条件：进程会因部署、扩缩容、信号（SIGTERM）等正常退出。
- 后置条件：收到停机信号后停止接收新请求，等待在途请求在宽限期内完成或安全中止，刷写缓冲、提交位点/位移、释放锁与连接、注销服务发现；超过宽限期才强制退出。
- 不变式：正常停机不丢失已接收且已确认的在途工作，不残留未释放的分布式锁或未注销的注册项导致后续流量黑洞。
- 验收标准：
  - 用例：发送 SIGTERM → 新请求被拒绝/转移，已接收请求在宽限期内完成，缓冲与位点被刷写。
  - 用例：持有分布式锁/服务注册的实例退出 → 锁被释放、注册被注销，不残留导致他人长期阻塞或流量打到死实例。

---
> 关联契约：与 failure-retry（超时/熔断释放）、concurrency（关闭竞态同步）、storage-durability（停机刷盘）、multi-tenancy（配额内资源回收）、observability（资源指标监控）互补；本契约聚焦"资源生命周期闭环与泄漏防护"这一横切维度。
