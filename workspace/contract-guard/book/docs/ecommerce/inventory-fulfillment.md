# 16 · 库存与履约契约（inventory-fulfillment）

> **适用领域**：`ecommerce`
> **适用场景**：库存扣减、预占 / 释放、秒杀、下单履约、发货。
> **原则**：只否决"超卖 / 负库存 / 预占泄漏 / 履约半成品"，不绑定具体扣减方案。

## 一句话理解

库存绝不能卖超、不能变负；预占的库存要么最终确认要么必须释放回去，订单履约不能停在"扣了库存没发货"的半截状态。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **INV-1** 不超卖、库存非负 | 可用库存 ≥ 0 恒成立 |
| **INV-2** 预占必有确认或释放 | 每个预占最终走向确认或超时释放，不泄漏 |
| **INV-3** 履约 Saga 可补偿 | 多步履约任一步失败可补偿回滚到一致状态 |

## 契约 INV-1：不超卖、库存非负

- **前置条件**：商品有有限库存，存在高并发抢购。
- **后置条件**：任意并发下扣减后可用库存 ≥ 0，不超卖。
- **不变式**：`可用库存 ≥ 0` 恒成立。
- **验收标准**：库存 1，1000 并发抢购 → 恰好 1 单成功，无负库存（依赖 [CONC-2](/universal/concurrency)）。

## 契约 INV-2：预占必有确认或释放

- **前置条件**：下单先预占库存，支付后再确认扣减。
- **后置条件**：每个预占要么被支付确认转为实扣，要么在超时后被释放归还，不长期挂占。
- **不变式**：不存在"既不确认也不释放"的僵尸预占；预占释放幂等。
- **验收标准**：下单不支付 → 超时后库存自动释放回可售；释放重复执行不超额归还（依赖 [IDEM-1](/universal/idempotency)）。

## 契约 INV-3：履约 Saga 可补偿

- **前置条件**：履约跨多步（扣库存 → 建发货单 → 通知物流）。
- **后置条件**：任一步失败，已完成步骤被补偿回滚，系统回到一致状态，不留半成品。
- **不变式**：履约流程的每一步都有对应补偿动作。
- **验收标准**：在中间步骤注入失败 → 已扣库存被补偿释放，无"扣了库存没发货"的孤儿订单（依赖 [DC-1](/universal/data-consistency)）。

## 现场提问骨架

- "高并发抢购靠什么保证不超卖、不出负库存？"
- "下单预占的库存，不付款会自动释放吗？释放会不会重复归还？"
- "履约中途某一步挂了，前面扣的库存怎么补偿回来？"

## 关联契约

- INV-1 依赖 [并发 CONC-2](/universal/concurrency)；INV-2 依赖 [幂等 IDEM-1](/universal/idempotency)；INV-3 依赖 [数据一致性 DC-1](/universal/data-consistency)。是多条通用契约的组合落地范例。
