# 导读：什么是「可验证契约」

> 一句话：契约不是"提醒你小心"，而是**一条可以被测试用例判定真假的下限**。

## 为什么需要它

做规格驱动开发（SDD）时，最常见的坑是：需求里写了"要保证不超卖""注意并发安全"，但这些话**无法验证**——它既不能写成测试，也不能在验收时判定"做到没做到"。结果就是看起来覆盖了，实际留了一堆暗坑。

`contract-guard` 的做法是：在调用 spec-kit 生成 spec/plan/tasks **之前**，先把相关的**可验证契约**注入到需求里，让产出从源头就带着"可被验收的下限"，而不是事后补救。

## 契约的四要素

每一条契约都由固定的四个部分组成。读懂这四要素，就读懂了所有契约：

| 要素 | 回答的问题 | 作用 |
| --- | --- | --- |
| **前置条件** | 什么场景下这条契约才适用？ | 划定适用边界，避免无差别套用 |
| **后置条件** | 操作结束后，系统必须呈现什么状态？ | 定义"做对了"长什么样 |
| **不变式** | 任意时刻都不能被打破的约束是什么？ | 最硬的红线，违反即出 bug |
| **验收标准** | 用什么用例能判定它真的成立？ | 把契约翻译成可执行的测试 |

::: tip 关键认知
**不变式**是契约里最值钱的一句。它描述的是"无论中间过程如何，这个等式/不等式恒成立"。比如幂等的不变式是"同一幂等键的副作用至多发生一次"，并发的不变式是"已分配量 ≤ 容量上限"。抓住不变式，就抓住了这条契约的灵魂。
:::

## 一条契约长什么样（以幂等为例）

```
契约 IDEM-1：写操作必须幂等
- 前置条件：客户端为每次逻辑操作携带稳定的幂等键，重试携带相同键。
- 后置条件：相同幂等键重复执行 N 次，最终状态与执行 1 次完全一致。
- 不变式：同一幂等键对应的副作用至多发生一次。
- 验收标准：并发提交同一幂等键 100 次 → 副作用计数恒为 1。
```

注意它**不绑定任何实现**：用 Redis、唯一索引还是状态机去重，都行。契约只否决"重复执行产生重复副作用"这个**结果**，不规定你怎么做。这是契约和"技术方案"的根本区别——**契约管下限，方案管怎么实现**。

## 契约之间会互相引用

契约不是孤立的。比如：

- 消息投递（MSG-1）的"消费幂等"直接**依赖**幂等（IDEM-1）；
- 库存履约（INV-1）的"不超卖"直接**依赖**并发（CONC-2）；
- 资金对账（FIN-2）的"回调不重复入账"同时**依赖**幂等（IDEM-1）和消息（MSG-1）。

所以越往后的领域契约，越是把前面的通用契约组合复用。**建议先吃透「通用基础」7 条，再看领域章节会轻松很多。**

## 库的组织原则（SSOT）

- **扁平存储**：每条契约在 `contracts/` 下只存一份文件，是唯一可信源（Single Source of Truth）。
- **多维标签**：每条契约打 `domains` 标签数组（可属于多个领域），靠标签复用，而不是把同一条契约在多个目录里复制。
- 本手册的章节分组，就是按 `domains` 主标签整理出来的一种**阅读视图**，方便你按领域学习。
