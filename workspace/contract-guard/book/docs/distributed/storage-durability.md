# 10 · 存储持久化与缓存一致性契约（storage-durability）

> **适用领域**：`storage` `universal` `distributed` `data-pipeline`
> **适用场景**：任何落盘存储、对象存储、以及"数据库 + 缓存"双写架构。
> **原则**：只否决"假成功未落盘 / 缓存长期脏 / 半截数据 / 孤儿文件 / 磁盘满崩溃"，不绑定具体存储引擎。

## 一句话理解

说"写成功了"就必须真落盘（重启还在）；用缓存就得想清楚缓存和源不一致的窗口有多大、怎么收敛。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **SD-1** 确认即持久 | 返回成功的写在崩溃 / 重启后仍可读 |
| **SD-2** 缓存一致性窗口有界 | 缓存与源的不一致时间有上界，防击穿/雪崩/穿透 |
| **SD-3** 原子写、无半截数据 | 写要么完整可见要么不可见，无半截 / 孤儿 |
| **SD-4** 容量与配额可控 | 磁盘 / 配额接近上限时优雅处理而非崩溃 |

## 契约 SD-1：确认即持久（durability）

- **前置条件**：写操作对外返回"成功"。
- **后置条件**：返回成功的数据已真正持久化（按需 fsync / 多副本确认），进程崩溃 / 重启后仍可读。
- **不变式**：已确认的写不会因崩溃丢失。
- **验收标准**：写成功后立即 kill 进程 → 重启后数据仍在。

## 契约 SD-2：缓存一致性窗口有界

- **前置条件**：存在缓存层（Redis / 本地缓存）与源存储。
- **后置条件**：源更新后缓存在有上界的时间内收敛；缓存击穿 / 雪崩 / 穿透有防护。
- **不变式**：缓存与源的不一致时间有明确上界；不存在永久脏缓存。
- **验收标准**：
  - 更新源 → 缓存在声明窗口内一致。
  - 热点 key 失效瞬间高并发 → 不击穿打垮源（有单飞 / 互斥重建）。
  - 查询不存在的 key → 不穿透打满源（有空值缓存 / 布隆过滤）。

## 契约 SD-3：原子写、无半截数据与孤儿

- **前置条件**：写入大对象 / 多段 / 关联文件与元数据。
- **后置条件**：写要么完整可见、要么完全不可见（临时文件 + rename 等）；不留半截文件、不留只有文件没元数据（或反之）的孤儿。
- **不变式**：外部可见状态中不存在半截 / 孤儿数据。
- **验收标准**：写入中途崩溃 → 不产生可被读到的半截对象，无孤儿引用。

## 契约 SD-4：容量与配额可控

- **前置条件**：存储有容量 / 配额上限。
- **后置条件**：接近上限时优雅拒绝新写并告警，不因磁盘满导致进程崩溃 / 数据损坏。
- **不变式**：磁盘满不破坏已有数据完整性。
- **验收标准**：注入磁盘满 → 新写被明确拒绝，已有数据完好，服务不崩。

## 现场提问骨架

- "返回成功是不是真落盘了？kill 进程后数据还在吗？"
- "缓存和数据库的不一致窗口多大？击穿 / 雪崩 / 穿透三件套都防了吗？"
- "写大文件中途崩溃会不会留半截或孤儿？磁盘满了会怎样？"
