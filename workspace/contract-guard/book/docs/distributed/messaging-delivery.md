# 08 · 消息投递语义契约（messaging-delivery）

> **适用领域**：`distributed` `ecommerce` `data-pipeline`
> **适用场景**：使用消息队列 / 事件总线进行异步通信、解耦、削峰。
> **原则**：只否决"消息丢 / 重复副作用 / 乱序致错 / 死信黑洞"，不绑定 Kafka / RocketMQ / RabbitMQ 等实现。

## 一句话理解

消息可能重复、可能乱序、可能堆积——消费端必须对这些都有确定的应对，确认成功的消息一条都不能丢。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **MSG-1** 消费幂等、副作用至多一次 | 同一消息无论投递几次，业务副作用至多一次 |
| **MSG-2** 确认即不丢、失败进死信 | 已确认消息不丢；处理失败的消息可追溯不黑洞 |
| **MSG-3** 顺序依赖被显式保证 | 有序需求的消息在同一分区按序消费 |

## 契约 MSG-1：消费幂等、副作用至多一次

- **前置条件**：投递语义为 at-least-once（绝大多数 MQ 的默认）。
- **后置条件**：同一条消息被重复投递时，消费端业务副作用至多发生一次。
- **不变式**：消费副作用以消息唯一标识去重，重复消费不产生重复业务结果。
- **验收标准**：同一消息重复投递 5 次 → 业务副作用计数为 1。

## 契约 MSG-2：确认即不丢、失败可追溯

- **前置条件**：消费端使用手动 ack / 提交位点。
- **后置条件**：只有处理成功才确认；处理失败的消息进入重试 / 死信队列（DLQ），不被静默丢弃。
- **不变式**：已 ack 的消息其业务效果已持久化；未成功处理的消息可被重新发现。
- **验收标准**：
  - 处理中崩溃（未 ack）→ 重启后该消息被重新投递处理。
  - 多次失败 → 进入死信队列并告警，不黑洞丢失。

## 契约 MSG-3：顺序依赖被显式保证

- **前置条件**：部分消息有顺序要求（如同一订单的状态流转）。
- **后置条件**：有序需求的消息按业务键路由到同一分区 / 队列，保证分区内有序消费。
- **不变式**：同一业务键的消息消费顺序与产生顺序一致。
- **验收标准**：同一订单的"创建→支付→发货"消息 → 消费端始终按此序处理。

## 现场提问骨架

- "消息重复投递时消费端幂等吗？靠什么去重？"
- "ack 是处理成功后才发吗？处理失败的消息进死信还是丢了？"
- "有顺序要求的消息，怎么保证不被乱序消费？"
