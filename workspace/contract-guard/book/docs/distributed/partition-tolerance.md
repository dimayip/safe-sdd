# 09 · 分区容错与共识契约（partition-tolerance）

> **适用领域**：`distributed` `multi-tenancy`
> **适用场景**：多节点集群、主从 / 多副本、需要在网络分区下保持正确性的系统。
> **原则**：只否决"脑裂双主 / 多数派失效仍写 / 因果序错乱"，不绑定 Raft / Paxos / ZAB 等具体算法。

## 一句话理解

网络一断成两半时，必须想清楚是保一致（CP）还是保可用（AP），并且绝不能出现"两个都自认为是主"在各写各的。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **PT-1** 明确 CAP 取舍并一致执行 | 分区期间的读写行为与声明的一致性级别相符 |
| **PT-2** 防脑裂、单一权威写 | 同一资源任意时刻至多一个分区可权威写 |
| **PT-3** 背压与流控不压垮自身 | 入流量超处理能力时优雅拒绝而非雪崩 |

## 契约 PT-1：明确 CAP 取舍并一致执行

- **前置条件**：系统由多节点组成，可能发生网络分区。
- **后置条件**：分区发生时，系统行为与对外声明的一致性级别（CP 拒绝写 / AP 降级可用）一致，不出现未声明的随机行为。
- **不变式**：分区期间的读写行为可预测，与声明相符。
- **验收标准**：注入网络分区 → 少数派侧按声明拒绝写或进入只读，不静默接受丢失写。

## 契约 PT-2：防脑裂、单一权威写

- **前置条件**：存在主节点 / leader 角色，需要选主。
- **后置条件**：任意时刻同一资源至多有一个权威写入方；旧主在失去多数派后不再接受写。
- **不变式**：不存在两个分区同时以"主"身份写同一资源（防双主 / 脑裂）。
- **验收标准**：主网络隔离 → 旧主停止写或被 fence，新主由多数派选出，无双写冲突。

## 契约 PT-3：背压与流控不压垮自身

- **前置条件**：上游可能突发高流量 / 慢消费。
- **后置条件**：入流量超过处理能力时，通过背压 / 限流优雅拒绝或排队，不导致 OOM / 雪崩。
- **不变式**：未确认 / 在途任务量有上界。
- **验收标准**：注入超额流量 → 系统限流降级，资源占用有界，不崩溃。

## 现场提问骨架

- "网络分区时你们是 CP 还是 AP？少数派那一侧还能写吗？"
- "怎么防脑裂？旧主失联后还能不能写？"
- "上游突发流量打过来，靠什么背压而不是被打挂？"
