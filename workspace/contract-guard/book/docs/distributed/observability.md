# 11 · 可观测性与监控告警契约（observability）

> **适用领域**：`observability` `engineering` `distributed` `multi-tenancy` `universal`
> **适用场景**：任何需要被运维、排障、度量 SLA 的线上系统。
> **原则**：只否决"出事看不见 / 告警漏报或风暴 / 日志泄密 / 链路断裂"，不绑定 Prometheus / OpenTelemetry 等实现。

## 一句话理解

出问题时要能"看得见、追得到、能定位"；告警既不能漏报（出事没人知道），也不能风暴（满屏告警等于没告警）。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **OB-1** 关键路径可贯穿追踪 | 一次请求在跨服务调用间可用统一 trace-id 串联 |
| **OB-2** 告警不漏报、不风暴 | 真故障必触发告警，且同源告警被聚合收敛 |
| **OB-3** 日志 / 指标脱敏合规 | 敏感信息不以明文进入日志 / 指标 |

## 契约 OB-1：关键路径可贯穿追踪

- **前置条件**：请求跨多个服务 / 组件。
- **后置条件**：一次请求在所有跨服务调用中携带统一 trace-id，可端到端串联耗时与错误。
- **不变式**：关键链路上 trace-id 不丢失、不断裂。
- **验收标准**：发起一次请求 → 在追踪系统中可看到完整调用链与各段耗时。

## 契约 OB-2：告警不漏报、不风暴

- **前置条件**：定义了关键 SLI / SLA 与告警规则。
- **后置条件**：真实故障必触发告警；同源大量告警被聚合 / 抑制，不淹没值班。
- **不变式**：核心故障不漏报；同一根因的告警被收敛为可处理的数量。
- **验收标准**：
  - 注入核心指标越界 → 在约定时间内触发告警。
  - 单一根因引发的连锁异常 → 告警被聚合，不刷屏。

## 契约 OB-3：日志 / 指标脱敏合规

- **前置条件**：日志 / 埋点可能包含用户敏感数据（手机号、token、身份证）。
- **后置条件**：敏感字段在落日志 / 上报前脱敏 / 掩码，不以明文存储。
- **不变式**：日志与指标中不出现明文敏感信息。
- **验收标准**：扫描日志样本 → 敏感字段均为掩码形式，无明文泄露。

## 现场提问骨架

- "一次请求跨了几个服务？能不能用一个 trace-id 把整条链路串起来看？"
- "核心故障会触发告警吗？一次根因爆发会不会告警风暴？"
- "日志里的手机号 / token 脱敏了吗？"
