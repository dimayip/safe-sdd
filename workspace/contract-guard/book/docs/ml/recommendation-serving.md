# 14 · 推荐系统服务质量契约（recommendation-serving）

> **适用领域**：`recommendation`
> **适用场景**：召回-排序-重排的推荐 / 信息流 / 搜索结果服务。
> **原则**：只否决"白屏无兜底 / 重复曝光 / 冷启动空 / 疲劳轰炸"，不绑定召回排序算法。

## 一句话理解

无论召回排序怎么波动，用户界面永远不能白屏——必须有兜底结果；同样的东西别反复推，新用户也得有得看。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **REC-1** 始终有非空兜底结果 | 任何情况下返回的推荐列表非空（有兜底） |
| **REC-2** 曝光去重与疲劳控制 | 同一用户短期内不重复曝光同一内容 |
| **REC-3** 冷启动有确定策略 | 新用户 / 新内容有明确的兜底召回 |

## 契约 REC-1：始终有非空兜底结果

- **前置条件**：召回 / 排序链路可能为空或超时。
- **后置条件**：任何情况下返回给前端的列表非空（热门 / 默认兜底），不白屏。
- **不变式**：对外响应的推荐结果数 ≥ 最小条数。
- **验收标准**：摘除主召回 / 排序超时 → 仍返回兜底列表，前端不白屏。

## 契约 REC-2：曝光去重与疲劳控制

- **前置条件**：同一用户多次刷新 / 翻页。
- **后置条件**：短期内已曝光内容不重复推送；对同类内容有疲劳度控制。
- **不变式**：同一用户在去重窗口内不重复曝光同一 item。
- **验收标准**：连续翻页 → 无重复 item；高频同类内容被降权。

## 契约 REC-3：冷启动有确定策略

- **前置条件**：存在无历史行为的新用户 / 无交互的新内容。
- **后置条件**：新用户有基于人群 / 热门的兜底召回；新内容有探索曝光机会。
- **不变式**：冷启动场景不返回空 / 不陷入零曝光黑洞。
- **验收标准**：新用户首次访问 → 返回合理的冷启动推荐而非空列表。

## 现场提问骨架

- "召回为空或排序超时的时候，前端是白屏还是有兜底？"
- "同一个用户翻页会不会反复看到同样的内容？疲劳怎么控？"
- "新用户、新内容的冷启动策略是什么？"
