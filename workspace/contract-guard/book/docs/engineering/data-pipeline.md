# 21 · 大数据批流处理契约（data-pipeline）

> **适用领域**：`data-pipeline` `distributed`
> **适用场景**：ETL、数仓、实时流处理（Flink/Spark）、批流一体。
> **原则**：只否决"重复计算 / 迟到乱序丢数 / 状态丢失 / 流批口径不一致"，不绑定具体引擎。

## 一句话理解

同一条输入在结果里只能体现一次（精确一次）；迟到和乱序的数据要有明确处理；流处理和批处理算出来的口径要一致。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **DP-1** 端到端精确一次 | 同一输入在结果汇中至多体现一次 |
| **DP-2** 迟到/乱序数据有确定处理 | 水位线 + 窗口策略明确，迟到数据不静默丢 |
| **DP-3** 流批口径一致 | 同一指标流式与批式结果在容差内一致 |

## 契约 DP-1：端到端精确一次（exactly-once）

- **前置条件**：作业可能因失败重启、重放数据。
- **后置条件**：通过 checkpoint + 幂等写 / 事务写，保证同一输入在结果汇中至多体现一次。
- **不变式**：作业重启 / 重放不导致结果汇重复累加。
- **验收标准**：作业中途 kill 后恢复 → 输出与无故障一次跑批结果一致，无重复。

## 契约 DP-2：迟到 / 乱序数据有确定处理

- **前置条件**：事件时间乱序到达，存在迟到数据。
- **后置条件**：用水位线（watermark）+ 窗口 + 允许迟到策略，明确定义迟到数据的归属与处理（更新 / 侧输出 / 丢弃并计数）。
- **不变式**：迟到数据不被静默丢弃；处理策略可观测。
- **验收标准**：注入乱序 / 迟到事件 → 按声明策略处理，迟到丢弃量有指标可见。

## 契约 DP-3：流批口径一致

- **前置条件**：同一指标既有实时流计算又有离线批计算。
- **后置条件**：相同口径的指标，流式结果与批式结果在容差内一致（lambda / kappa 对齐）。
- **不变式**：流批同口径指标的差异有上界。
- **验收标准**：对同一时间窗口的指标做流批对账 → 差异在容差内。

## 现场提问骨架

- "端到端是 at-least-once 还是 exactly-once？作业重启会不会重复累加？"
- "迟到、乱序的数据怎么处理？水位线和窗口策略是什么？丢弃的有统计吗？"
- "实时大盘和离线报表的口径对得上吗？"
