# 23 · 音视频传输与实时通信契约（media-streaming）

> **适用领域**：`media-streaming` `distributed`
> **适用场景**：直播、RTC / WebRTC、RTMP 推流、HLS 拉流、音视频编解码。
> **原则**：只否决"协商失败黑屏 / 弱网卡死 / 音画不同步 / 码率不自适应"，不绑定具体编解码器。

## 一句话理解

收发双方的编解码能力要协商出交集；弱网丢包时宁可降质也不能卡死或崩溃；声音和画面要对得上。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **MS-1** 编解码能力协商取交集 | 双方按协商出的公共能力通信，不会黑屏 |
| **MS-2** 弱网降质不崩溃 | 丢包 / 抖动下自适应降码率，不卡死不崩 |
| **MS-3** 音画同步在容差内 | 音视频播放时间差在可接受容差内 |

## 契约 MS-1：编解码能力协商取交集

- **前置条件**：收发端支持的编解码 / 分辨率 / 协议可能不同。
- **后置条件**：连接建立时协商出双方都支持的公共能力集，按交集通信。
- **不变式**：实际使用的编解码参数 ∈ 双方能力交集，不会出现一方解不了的流。
- **验收标准**：不同能力的端互连 → 协商成功并正常出画，无黑屏 / 无法解码。

## 契约 MS-2：弱网降质不崩溃

- **前置条件**：网络存在丢包、抖动、带宽波动。
- **后置条件**：通过 ABR 自适应码率 / FEC / 重传 / 抖动缓冲，弱网下平滑降质，不卡死、不崩溃。
- **不变式**：可用带宽下降时主动降码率，码流不超过实际带宽导致堆积崩溃。
- **验收标准**：注入丢包 / 限速 → 画质平滑下降并恢复，无长时间卡顿 / 崩溃。

## 契约 MS-3：音画同步在容差内

- **前置条件**：音频与视频分轨传输、独立解码。
- **后置条件**：播放时音视频按时间戳对齐，唇音同步偏差在可接受容差内（通常几十毫秒级）。
- **不变式**：音视频播放时间差 ≤ 同步容差。
- **验收标准**：长时间播放 / 弱网恢复后 → 音画同步偏差保持在容差内，不累积漂移。

## 现场提问骨架

- "收发端编解码不一样时怎么协商？会不会黑屏 / 解不了码？"
- "弱网丢包时是自适应降码率还是直接卡死 / 崩溃？"
- "音画同步怎么保证？长时间播放会不会越漂越远？"
