# 19 · 防滥用与安全边界契约（abuse-security）

> **适用领域**：`universal` `ecommerce` `recommendation` `engineering` `distributed` `realtime-iot`
> **适用场景**：对外接口、鉴权授权、防重放、防刷量 / 薅羊毛、风控。
> **原则**：只否决"重放攻击 / 越权 / 被刷穿 / 默认放行"，不绑定具体风控引擎。

## 一句话理解

凭证用过一次就作废（防重放）；只能访问授权范围内的东西（防越权）；接口要扛得住刷量；拿不准时默认拒绝而非放行。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **SEC-1** 防重放、凭证至多用一次 | 已消费的 nonce / 凭证不可被二次接受 |
| **SEC-2** 鉴权默认拒绝、最小权限 | 未明确授权的访问一律拒绝 |
| **SEC-3** 限流防刷有效 | 异常高频请求被限流 / 风控拦截 |

## 契约 SEC-1：防重放、凭证至多用一次

- **前置条件**：请求带签名 / nonce / 一次性凭证。
- **后置条件**：同一 nonce / 凭证被重复提交时，第二次起被拒绝。
- **不变式**：已消费的 nonce / 一次性凭证在有效期内不可再次接受。
- **验收标准**：抓包重放同一已签名请求 → 第二次被拒绝。

## 契约 SEC-2：鉴权默认拒绝、最小权限

- **前置条件**：资源 / 接口有访问控制需求。
- **后置条件**：未明确授权的访问一律拒绝（白名单 / 默认 deny）；权限按最小必要授予。
- **不变式**：不存在"忘了配权限就默认放行"的接口。
- **验收标准**：
  - 越权访问他人资源 → 被拒绝（403）。
  - 新增接口未配权限 → 默认拒绝而非默认开放。

## 契约 SEC-3：限流防刷有效

- **前置条件**：接口可能被高频刷 / 薅羊毛。
- **后置条件**：异常高频 / 异常模式请求被限流或风控拦截。
- **不变式**：单一用户 / IP 的请求速率有上界。
- **验收标准**：模拟刷量 → 超阈值被限流 / 验证码 / 风控拦截，核心资源不被刷穿。

## 现场提问骨架

- "防重放、防越权、防刷量这三道分别怎么挡？"
- "新加的接口忘了配权限会默认放行还是默认拒绝？"
- "薅羊毛 / 刷量场景，限流和风控怎么生效？"
