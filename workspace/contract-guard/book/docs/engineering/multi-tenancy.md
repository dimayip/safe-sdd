# 20 · 多租户与资源隔离契约（multi-tenancy）

> **适用领域**：`multi-tenancy` `engineering` `distributed`
> **适用场景**：SaaS、多租户平台、共享集群上的资源配额与隔离。
> **原则**：只否决"数据串台 / 噪声邻居拖垮全局 / 配额失控超售"，不绑定隔离实现（行级 / schema 级 / 物理级）。

## 一句话理解

租户之间的数据绝不能串台；任何一个租户都不能因为自己的暴用，把别人和整个平台拖垮。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **MT-1** 租户数据强隔离、不串台 | 任一数据访问的作用域 ⊆ 当前租户 |
| **MT-2** 配额与噪声邻居受控 | 单租户资源占用有上界，故障半径受限 |
| **MT-3** 配额不超售失控 | 已分配配额之和在物理容量约束内可管理 |

## 契约 MT-1：租户数据强隔离、不串台

- **前置条件**：多个租户共享同一套服务 / 数据库。
- **后置条件**：每个查询 / 写入都强制带租户作用域，A 租户无法读写 B 租户数据。
- **不变式**：任一数据访问的作用域 ⊆ 当前请求的租户。
- **验收标准**：构造缺失 / 伪造租户标识的请求 → 被拒绝，不返回跨租户数据。

## 契约 MT-2：配额与噪声邻居受控

- **前置条件**：租户共享 CPU / 连接 / 限流额度等资源。
- **后置条件**：单租户的资源消耗有上限，超限被限流，不拖垮其他租户。
- **不变式**：单租户故障 / 暴用的影响半径被限制在该租户内。
- **验收标准**：单租户发起暴力流量 → 仅自身被限流降级，其他租户 SLA 不受影响。

## 契约 MT-3：配额不超售失控

- **前置条件**：给租户分配资源配额。
- **后置条件**：配额分配可管理；即便超售也有抢占 / 降级策略，不导致整体不可用。
- **不变式**：已承诺配额与物理容量的关系可监控、可治理。
- **验收标准**：配额接近物理上限 → 触发告警 / 拒绝新分配，不静默超售崩溃。

## 现场提问骨架

- "租户数据怎么保证不串台？每个查询都强制带租户隔离了吗？"
- "一个租户疯狂打流量，会不会把其他租户和整个平台拖垮？"
- "配额会不会超售？超售了有没有降级 / 抢占策略？"
