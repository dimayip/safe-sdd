# 22 · 实时系统与 IoT 契约（realtime-iot）

> **适用领域**：`realtime-iot` `distributed`
> **适用场景**：IoT 设备接入、实时控制、时序采集、边缘计算、指令下发。
> **原则**：只否决"时延超限 / 断连丢数据 / 指令重复执行 / 用陈旧数据决策"，不绑定 MQTT 等协议。

## 一句话理解

实时路径必须在硬时延上界内完成；设备会离线断连，要有缓存补传；下发的控制指令必须幂等；别拿过期的旧数据当成现状去决策。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **RT-1** 实时路径时延有硬上界 | 关键实时处理耗时 ≤ 约定硬上界 |
| **RT-2** 断连可恢复、数据不丢 | 设备离线期间数据缓存，恢复后补传去重 |
| **RT-3** 下行指令幂等、数据新鲜 | 同一指令重复下发至多执行一次；决策基于新鲜数据 |

## 契约 RT-1：实时路径时延有硬上界

- **前置条件**：存在有实时性要求的处理 / 控制环路。
- **后置条件**：关键实时路径的处理时延有硬上界，超时有明确兜底（保持上一状态 / 安全停机）。
- **不变式**：实时路径时延 ≤ 约定上界，超时被检测。
- **验收标准**：压测实时链路 → 时延 P99 在硬上界内；超时触发兜底而非阻塞。

## 契约 RT-2：断连可恢复、数据不丢

- **前置条件**：设备网络不稳定，会离线 / 断连。
- **后置条件**：设备离线期间本地缓存数据，恢复连接后补传；服务端按时序去重合并。
- **不变式**：离线期间产生的数据在恢复后可被补齐，不静默丢失。
- **验收标准**：断网一段时间再恢复 → 期间数据补传成功，无重复、无丢失（去重依赖 [IDEM-1](/universal/idempotency)）。

## 契约 RT-3：下行指令幂等、数据新鲜

- **前置条件**：向设备下发控制指令；用采集数据做决策。
- **后置条件**：同一指令重复下发至多执行一次；用于决策的数据在新鲜度窗口内，过期数据不参与控制。
- **不变式**：指令执行幂等；决策数据的时间戳在新鲜度上界内。
- **验收标准**：
  - 重复下发同一指令 → 设备至多执行一次。
  - 喂入过期采样数据 → 被识别为陈旧并拒绝 / 降级，不用于实时控制。

## 现场提问骨架

- "实时路径有硬时延上界吗？超时了是兜底还是一直阻塞？"
- "设备断网期间的数据怎么办？恢复后补传会不会重复或丢失？"
- "下发的指令重复到达会不会执行两次？决策用的数据会不会是过期的？"

## 关联契约

- RT-2/RT-3 的去重依赖 [幂等 IDEM-1](/universal/idempotency)；数据新鲜度依赖 [时间正确性 TIME](/universal/time-correctness)。
