# 18 · 配置变更与发布安全契约（config-release-safety）

> **适用领域**：`engineering`
> **适用场景**：配置下发、灰度发布、上线部署、依赖编排。
> **原则**：只否决"配置一改全崩 / 不能回滚 / 循环依赖死锁"，不绑定 CI/CD 工具。

## 一句话理解

任何上线和配置变更都必须能灰度、能一键回滚；生效前要校验，依赖关系不能成环。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **CRS-1** 变更可灰度、可回滚 | 任意已发布变更都有可执行的回滚路径 |
| **CRS-2** 配置生效前校验 | 非法配置在生效前被拦截，不进入运行态 |
| **CRS-3** 依赖编排无环 | 启动 / 发布依赖构成 DAG，无循环依赖 |

## 契约 CRS-1：变更可灰度、可回滚

- **前置条件**：发布会影响线上行为。
- **后置条件**：变更支持灰度放量；出问题可在有限时间内一键回滚到上一稳定版本。
- **不变式**：任意已发布变更都有可执行的回滚路径。
- **验收标准**：灰度发现异常 → 一键回滚后服务恢复，状态与发布前一致。

## 契约 CRS-2：配置生效前校验

- **前置条件**：配置可被动态修改并下发。
- **后置条件**：配置在生效前经过 schema / 取值 / 依赖校验，非法配置被拒绝，不进入运行态。
- **不变式**：运行态配置始终是经过校验的合法配置。
- **验收标准**：下发非法配置 → 被拦截并告警，线上仍用旧的合法配置（依赖 [BN-1](/universal/boundary-nullness)）。

## 契约 CRS-3：依赖编排无环

- **前置条件**：服务 / 模块 / 配置之间有启动或加载依赖。
- **后置条件**：依赖关系构成有向无环图（DAG），按拓扑序初始化。
- **不变式**：不存在循环依赖导致启动死锁。
- **验收标准**：依赖图做环检测 → 无环；引入环 → 在构建 / 启动期被检出报错。

## 现场提问骨架

- "这次发布能灰度吗？出问题能不能一键回滚？回滚验证过吗？"
- "配置生效前校验吗？下发一个错配置会不会直接把线上搞挂？"
- "模块 / 服务之间会不会有循环依赖导致启动卡死？"

## 关联契约

- CRS-1 的回滚与 [资源生命周期 RES-3](/universal/resource-lifecycle) 的优雅停机协同；与 [接口兼容 API](/engineering/api-compatibility) 共同保障发布安全。
