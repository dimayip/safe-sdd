# 17 · 接口兼容性契约（api-compatibility）

> **适用领域**：`engineering` `distributed`
> **适用场景**：对外 / 跨服务 API、schema 演进、版本升级。
> **原则**：只否决"破坏性变更悄悄上线 / 老调用方被打挂"，不绑定 REST / gRPC / GraphQL。

## 一句话理解

接口可以演进，但不能让那些没改代码的老调用方突然调不通——破坏性变更必须升主版本。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **API-1** 同主版本向后兼容 | 旧调用方不改代码仍能正常调用同主版本接口 |
| **API-2** 破坏性变更升主版本 | 任何破坏性变更通过新版本号隔离，老版本平滑下线 |

## 契约 API-1：同主版本向后兼容

- **前置条件**：接口有存量调用方，需要迭代。
- **后置条件**：在同一主版本内，只做兼容性变更（加可选字段、加接口），不删字段 / 不改语义 / 不收紧约束。
- **不变式**：旧调用方不修改代码即可正常调用同主版本接口。
- **验收标准**：用旧版本契约测试（CDC）跑新服务 → 全部通过。

## 契约 API-2：破坏性变更升主版本、老版本平滑下线

- **前置条件**：需要做破坏性变更（删字段、改类型、改语义）。
- **后置条件**：破坏性变更发布为新主版本，新旧版本并存一段时间，老版本有明确下线公告与期限。
- **不变式**：同一时刻调用方总能找到一个可用的兼容版本。
- **验收标准**：发布 v2 破坏性变更 → v1 仍可用至公告期，调用方平滑迁移，无强制断流。

## 现场提问骨架

- "这次接口变更对存量调用方兼容吗？删了字段还是改了语义？"
- "破坏性变更升主版本了吗？老版本什么时候下线、有没有通知到调用方？"
- "有没有用消费者驱动契约测试（CDC）卡兼容性？"
