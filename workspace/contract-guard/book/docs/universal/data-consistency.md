# 03 · 数据一致性契约（data-consistency）

> **适用领域**：`universal` `distributed` `ecommerce` `ml-serving` `data-pipeline`
> **适用场景**：一次业务操作跨多个存储 / 表 / 服务，需要它们的状态相互吻合。
> **原则**：只否决"持久化半成品 / 对不上账"，不绑定一致性实现（事务 / 2PC / Saga / 对账皆可）。

## 一句话理解

一笔操作要动好几张表，结束后它们要么都改了、要么都没改，绝不能"只改了一半还永久停在那"。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **DC-1** 跨资源写全生效或可收敛一致 | 业务勾稽关系操作后恒成立，未完成有 pending 标记 |
| **DC-2** 读到的状态有明确一致性语义 | "最终一致"数据有可测量的收敛上界 |

## 契约 DC-1：跨资源写要么全部生效要么可收敛到一致

- **前置条件**：一次业务操作需要修改 ≥2 个独立可持久化状态（如订单表 + 库存表 + 账务表）。
- **后置条件**：操作结束后，要么所有状态都体现该操作，要么通过补偿 / 重试最终都不体现；不存在"只改了一半且永久停留"的状态。
- **不变式**：业务层定义的勾稽关系（如 订单金额 = Σ明细金额）在操作完成后恒成立，未完成时有明确的中间态标记（pending）而非伪终态。
- **验收标准**：
  - 第 2 个写失败 → 第 1 个写被回滚或被补偿，无残留半成品。
  - 对任意时间点快照做勾稽校验 → 终态记录全部对账平衡。

## 契约 DC-2：读到的状态有明确一致性语义

- **前置条件**：存在主从 / 缓存 / 多副本，读写可能走不同节点。
- **后置条件**：对外承诺的一致性级别（强一致 / 读己之写 / 最终一致 + 收敛上界）被明确声明且可验证。
- **不变式**：声明为"最终一致"的数据有可测量的收敛时间上界，超时即告警。
- **验收标准**：
  - 写后立即读（同会话）→ 满足读己之写。
  - 注入副本延迟 → 在声明上界内收敛，超界触发告警。

## 现场提问骨架

- "这笔操作跨了几个存储？中途某一步失败，已写的怎么回滚或补偿？"
- "你们承诺的是强一致还是最终一致？最终一致的收敛上界是多少？"
- "能不能拿日终快照做勾稽校验来验收？"
