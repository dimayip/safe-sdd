# 06 · 资源生命周期与泄漏契约（resource-lifecycle）

> **适用领域**：`universal` `engineering` `distributed` `storage` `multi-tenancy`
> **适用场景**：任何使用文件句柄、连接、线程 / 协程、内存、锁等需要"用完归还"的资源的长跑服务。
> **原则**：只否决"泄漏 / use-after-close / 停机丢请求"，不绑定 RAII / defer / try-finally 等具体机制。

## 一句话理解

凡是"借来"的资源，无论成功还是异常，都必须"还回去"；进程要停时，手头在干的活要收好尾。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **RES-1** 获取与释放成对、异常也释放 | 任一执行路径上，已获取资源最终都被释放 |
| **RES-2** 池化资源借还守恒 | 借出数 ≤ 池容量，归还后状态干净 |
| **RES-3** 优雅停机不丢在途请求 | 收到停机信号后，在途请求处理完或被安全转移 |

## 契约 RES-1：资源获取与释放成对，异常路径也释放

- **前置条件**：代码获取了需显式释放的资源（fd / 连接 / 锁 / 内存）。
- **后置条件**：无论正常返回还是抛异常 / 提前 return，资源都被释放，不泄漏。
- **不变式**：任一执行路径上，已获取的资源最终都被释放；不存在 use-after-close / double-free。
- **验收标准**：
  - 在持有资源时注入异常 → 资源仍被释放（fd 数、连接数回落）。
  - 长时间压测 → 句柄 / 内存曲线平稳，无单调上涨。

## 契约 RES-2：池化资源借还守恒

- **前置条件**：使用连接池 / 线程池等池化资源。
- **后置条件**：借出的资源必被归还；归还的资源状态干净（无残留事务 / 脏数据）。
- **不变式**：`借出数 ≤ 池容量`；不存在只借不还导致池耗尽。
- **验收标准**：高并发借还 → 池不被耗尽，空闲数可回到初值；异常路径也归还。

## 契约 RES-3：优雅停机不丢在途请求

- **前置条件**：服务会被重启 / 缩容（收到 SIGTERM）。
- **后置条件**：收到停机信号后，停止接收新请求，已接收的在途请求处理完或安全转移，再退出。
- **不变式**：停机过程中不强杀在途请求导致状态不一致。
- **验收标准**：发送 SIGTERM → 在途请求全部完成 / 安全失败，无半成品写入。

## 现场提问骨架

- "异常路径上资源会不会漏释放？压测时句柄 / 内存稳不稳？"
- "连接池借出的连接一定会归还吗？会不会被打满？"
- "滚动发布时，正在处理的请求会被直接杀掉吗？"
