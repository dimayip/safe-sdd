# 02 · 并发竞态契约（concurrency）

> **适用领域**：`universal` `distributed` `ecommerce`
> **适用场景**：多请求 / 多线程 / 多实例同时读改写同一份共享状态。
> **原则**：只否决"丢更新 / 脏读 / 双花"，不绑定锁实现（乐观锁 / 悲观锁 / CAS / 串行化皆可）。

## 一句话理解

多个人同时改同一个数，最终结果必须等于"按某个顺序一个个改"的结果，不能有谁的修改被悄悄覆盖。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **CONC-1** 读-改-写无丢更新 | 每次成功修改都基于读取时的真实版本 |
| **CONC-2** 临界资源不被双重占用 | 已分配量 ≤ 容量上限，恒成立 |
| **CONC-3** 无死锁 / 可终止 | 资源获取顺序全局一致，或等待有上界 |

## 契约 CONC-1：共享状态的读-改-写必须无丢更新

- **前置条件**：存在被并发修改的共享资源（库存、余额、计数器、状态字段）。
- **后置条件**：N 个并发"读-改-写"全部成功提交后，最终值等于按某个串行序逐个应用的结果，无更新被覆盖丢失。
- **不变式**：对共享资源的每次成功修改都基于其被读取时的真实版本（版本号 / CAS 校验），版本不匹配则该次修改失败重来。
- **验收标准**：
  - 初始计数 0，100 并发各 +1 → 最终值恒为 100。
  - 基于过期版本提交 → 被拒绝并触发重试，不静默覆盖。

## 契约 CONC-2：临界资源不被双重占用（防双花 / 超卖前置）

- **前置条件**：资源有可量化的容量上限（库存数、座位、配额）。
- **后置条件**：任意并发场景下，已分配总量不超过容量上限。
- **不变式**：`已分配量 ≤ 容量上限` 恒成立，不存在中间态突破上限。
- **验收标准**：库存 1，100 并发抢购 → 恰好 1 个成功，其余明确失败，不出现负库存。

## 契约 CONC-3：无死锁 / 可终止

- **前置条件**：一次操作可能持有多把锁 / 多个资源。
- **后置条件**：不存在两两互相等待导致永久阻塞；锁等待有超时或有序加锁规避环路。
- **不变式**：资源获取顺序全局一致，或任一等待有上界。
- **验收标准**：交叉加锁压测无永久阻塞；超时后能回滚并返回明确错误。

## 现场提问骨架

- "高并发改同一份数据时，靠什么保证没有丢更新？用了版本号 / CAS 吗？"
- "能不能用『100 并发各 +1，最终恰好 100』来验收？"
- "多把锁的获取顺序统一了吗？怎么避免死锁？"
