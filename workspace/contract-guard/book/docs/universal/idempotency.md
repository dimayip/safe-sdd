# 01 · 幂等性契约（idempotency）

> **适用领域**：`universal` `distributed` `ml-serving` `recommendation` `ecommerce` `engineering` `data-pipeline` `realtime-iot`
> **适用场景**：任何"同一请求可能被重复提交 / 重复投递 / 重试"的写操作。
> **原则**：只否决"重复执行产生重复副作用"，不绑定具体去重实现（Redis / 唯一索引 / 状态机皆可）。

## 一句话理解

同一个逻辑操作做一次和做一百次，对系统的影响必须**完全一样**。

## 契约速览

| 契约 | 不变式（红线） |
| --- | --- |
| **IDEM-1** 写操作必须幂等 | 同一幂等键对应的副作用至多发生一次 |
| **IDEM-2** 幂等键的过期与冲突可判定 | 幂等键 → 操作语义 的绑定在有效期内唯一 |

## 契约 IDEM-1：写操作必须幂等

- **前置条件**：客户端为每次"逻辑操作"携带稳定的幂等键（idempotency key），且同一逻辑操作的重试携带相同键。
- **后置条件**：以相同幂等键重复执行 N 次（N≥2），系统最终状态与执行 1 次完全一致；重复请求返回首次执行的结果而非再次产生副作用。
- **不变式**：任意时刻，同一幂等键对应的副作用（扣款 / 建单 / 发消息等）至多发生一次。
- **验收标准**：
  - 并发提交同一幂等键 100 次 → 副作用计数恒为 1。
  - 首次成功后重放 → 返回首次结果，无第二次副作用。
  - 首次执行中途失败后重试 → 不产生半成品状态，重试后结果唯一确定。

## 契约 IDEM-2：幂等键的过期与冲突可判定

- **前置条件**：幂等键有明确的有效期与作用域（哪个接口 / 哪个租户）。
- **后置条件**：过期后复用同一键不会错误命中旧结果；不同语义请求复用同一键时必须被拒绝而非静默覆盖。
- **不变式**：幂等键 → 操作语义 的绑定在有效期内唯一。
- **验收标准**：相同键 + 不同请求体 → 返回冲突错误（如 409），不执行。

## 现场提问骨架

> 把不变式改成疑问句，直接问对方：

- "重复提交 / 回调重投时，靠什么保证只产生一次副作用？幂等键是什么？"
- "同一个幂等键配上不同的请求体，你们是覆盖还是拒绝？"
- "能不能用『同一键并发 100 次，副作用计数为 1』这个用例验收？"
